Kiro + OpenClaw
AWS 运维助理

用自然语言管理 AWS 资源。Kiro 负责思考和编排，OpenClaw 负责执行和交互。 通过 SSM 加密隧道连接，多层权限控制，全链路可审计——安全、可控、高效。

Kiro IDE Kiro CLI OpenClaw AWS MCP SSM Terraform CloudTrail

协作架构

完整的协作链路，每一层都有明确的职责和安全边界：

开发者 → Kiro IDE / CLI → SSM 加密隧道 → OpenClaw Agent → MCP / Skills → AWS 服务
  • 交互层：开发者在 Kiro 中用自然语言描述意图
  • 传输层：SSM 端口转发，IAM 认证，全程加密，零公网暴露
  • 执行层：OpenClaw Agent 调用 MCP 工具执行操作
  • 资源层：EC2、VPC、RDS、S3、Bedrock 等 AWS 服务

安全与可控性

AI 驱动运维最大的顾虑是"会不会失控"。这套方案从架构层面解决了这个问题：

🛡️ Human-in-the-Loop

资源变更类操作的 MCP autoApprove 设为空，每次操作暂停等待人工确认。AI 提议，人决策——比直接敲命令多了一层审查缓冲。

🔒 网络零暴露

OpenClaw 绑定 localhost，安全组入站规则为空。所有访问通过 SSM 加密隧道，不需要 SSH 密钥，不需要 VPN，攻击面为零。

🔑 权限分层

IAM 最小权限 → MCP 操作审批 → OpenClaw sandbox 隔离 → tools.allow/deny 白名单 → SCP 组织级管控。五层纵深防御，逐级拦截。

📋 全链路审计

CloudTrail 记录 API 调用，SSM 记录 session 日志，OpenClaw 记录对话历史，MCP 记录工具调用。出了问题可以完整回溯。

典型运维场景

🚀 部署新资源

「帮我在 us-east-1 创建一个 t3.medium 的 EC2」→ Kiro 编排 → OpenClaw 调用 Terraform MCP 生成 IaC 代码 → 人工审查 → 执行部署

🔧 故障排查

「OpenClaw 没响应了」→ Kiro 通过 SSM 让 OpenClaw 检查服务状态、读日志、定位问题 → 生成修复方案 → 确认后执行

🔄 模型切换

「把 AI 模型换成 Claude Sonnet 4.6」→ OpenClaw 修改配置 → 重启 Gateway → 验证新模型响应正常

🛡️ 安全审计

「检查安全组配置是否合规」→ OpenClaw 查询安全组规则 → Checkov 合规检查 → 输出审计报告

最佳实践

  • IaC 优先：所有基础设施变更通过 Terraform/CloudFormation 代码实现，不直接用 CLI 修改
  • 最小权限：IAM Role 只授予当前任务所需的最小权限，定期审查和收缩
  • 操作审批：资源变更类 MCP 工具保持 autoApprove 为空，每次人工确认
  • 定期审计：定期检查 CloudTrail 日志、安全组规则、IAM 策略
  • 沙箱执行：OpenClaw Agent 在 sandbox 模式下运行，限制文件系统和网络访问
  • 版本控制：所有配置文件和 IaC 代码纳入 Git 管理
